# Benchmarking Graphenium

## Why Benchmark Tokens?

LLM context windows are expensive — both in monetary cost and in reasoning quality.
Every token that isn't structurally relevant consumes space the model could use for
understanding. Graphenium's core value proposition is **reducing the number of tokens
needed for structural understanding** by extracting a compact knowledge graph from
source code.

Without Graphenium, a language model asked to reason about a codebase typically needs:

- The full source tree ingested as context
- Multiple rounds of file-reading tool calls to find relevant symbols
- Long reasoning chains to reconstruct relationships the model must infer from scratch

With Graphenium, the same task starts from a pre-computed graph that captures symbols,
dependencies, call relationships, and community structure. The model can answer
structural questions with a fraction of the input and output tokens.

Benchmarking quantifies this advantage.

---

## Methodology

Run a representative set of tasks **with** and **without** Graphenium (or with different
Graphenium configurations), and record the following metrics for each trial:

| Metric | Description |
|--------|-------------|
| **Input tokens** | Total tokens consumed by the prompt, including files, tool results, and conversation history needed to reach a correct answer |
| **Output tokens** | Total tokens generated by the model in reasoning and final answer |
| **Time to correct answer** | Wall-clock time from task start to correct answer delivery |
| **Plan correctness** | Whether the model's structural reasoning or plan was correct (pass/fail or 0–1 score) |

### Suggested comparison matrix

| Condition | Graph available? | Typical setup |
|-----------|-----------------|---------------|
| Baseline (no Graphenium) | No | Model reads source files directly via file-reading tools |
| Graphenium query | Yes | Model receives graph query results; no raw file reads |
| Graphenium full | Yes | Model receives graph query results *and* selected source files |

> **Pro tip**: Run each condition 3–5 times and report the median, because LLM
> outputs vary across runs.

## How to Run Your Own Benchmarks

### Prerequisites

- `gm` (Graphenium CLI) installed and in `$PATH`
- A graph built for the target repository:
  ```bash
  cd /path/to/repo
  gm run .
  ```

### Manual benchmark

```bash
# 1. Measure a query's character output
gm query "your query here" --budget 2000 | wc -c

# 2. (Optional) Time it
time gm query "your query here" --budget 2000 > /dev/null

# 3. Compare with baseline: what would it take to answer the same
#    question by reading source files directly?
```

### Automated benchmark suite

```bash
bash scripts/run_benchmarks.sh
```

This script runs a standard set of queries, asserts each stays under a character
budget, and prints a pass/fail summary.

### Adding a new benchmark scenario

1. Add a query to `scripts/run_benchmarks.sh`
2. Add a row to the **Benchmark Scenarios** table above
3. Run the baseline (no Graphenium) version of the same task to collect comparison data

## Interpreting Results

### What good looks like ✅

| Signal | What it means |
|--------|---------------|
| Query output < 10,000 chars (~2,500 tokens) | The graph is returning terse, high-signal results — ideal |
| Token reduction > 10× vs baseline | Graphenium is proving its value for this scenario |
| Time-to-correct-answer under 30 seconds | The graph query is fast enough for interactive use |
| Plan correctness ≥ 80% with Graphenium | The graph provides accurate structural information |

### Concerning numbers ⚠️

| Signal | What it means |
|--------|---------------|
| Query output > 50,000 chars | The budget is too high or the graph is too noisy; try lowering `--budget` |
| Token reduction < 2× vs baseline | For this particular task, the graph may not be adding much over raw file access |
| Time-to-correct-answer > 2 minutes | Either the graph is very large, or the query is too broad — consider narrowing scope |
| Plan correctness < 50% | The graph may be stale or have unresolved references; run `gm doctor` to check graph quality |
| Multiple queries exceed budget | The budget thresholds in `run_benchmarks.sh` may need tuning for your repository size |

### When to adjust budgets

Budgets in the benchmark script are set conservatively for a medium-sized repository
(10k–50k LOC). For larger repositories, you may need to increase the character budgets.
For smaller ones, tighten them to keep benchmarks meaningful.
